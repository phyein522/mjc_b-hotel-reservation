# 호텔 요금(Rate) 관리 기능 작업 지시서

대상: agy
기술 스택: Java 21 / Gradle / Spring Boot 4.1.0 (Web, Security) / MySQL / Spring Data JPA + MyBatis(기존 작성분, 신규 작성 금지) / Swagger UI / Lombok

---

## ⚠️ 작업 범위 제한 (절대 규칙, 예외 없음)

- 새로 만드는 코드는 전부 **`rates` 패키지(폴더) 안에서만** 작성한다. Entity, DTO, Controller, Service, Repository 전부 이 폴더 하위에만 위치시킨다.
- **`rates` 폴더 밖의 파일은 절대 수정, 삭제, 이동하지 않는다.** 이건 팀원들과 같은 저장소를 공유하며 각자 작업 후 머지(merge)할 예정이기 때문이다. 다른 폴더를 건드리면 팀원의 작업과 충돌(merge conflict)이 나거나 팀원 코드를 덮어써버릴 수 있다.
- 기존 Entity(`rooms`, `hotels`, `bookings` 등)를 참조해야 할 때도 그 파일을 직접 수정하지 않는다. `rates` 패키지 안에서 FK 값(예: `hotelId`, `roomId`)만 필드로 들고 있고, 필요하면 `@ManyToOne` 등으로 참조만 한다.
- DB 스키마 변경(ALTER TABLE 등)은 이미 ERDCloud에 반영 완료된 상태이므로, 코드에서 별도로 DDL을 실행하거나 마이그레이션 파일을 새로 만들 필요 없다. Entity 매핑만 하면 된다.
- 작업 중 "이 부분은 다른 폴더도 고쳐야 동작한다"는 상황이 생기면, 임의로 고치지 말고 먼저 사용자에게 알린다.

## 0. 작업 시작 전 반드시 먼저 할 일

이 문서에 나오는 클래스명, 필드명, API 경로는 전부 "참고용 제안"이다. 실제 작업 시작 전에 아래 순서를 반드시 지킨다.

1. 저장소(mjc_b-hotel-reservation)에서 팀원들이 이미 작성한 기존 패키지 구조, Entity 클래스, DTO, Controller 네이밍 규칙을 먼저 읽는다.
2. 기존 코드에 이미 있는 스타일(예: 패키지 depth, 어노테이션 조합, 응답 포맷, 예외 처리 방식, Repository/Service 분리 패턴)을 그대로 따른다.
3. 이 문서의 제안과 기존 코드 스타일이 충돌하면 **기존 코드 스타일을 우선**한다. 이 문서는 "무엇을 만들어야 하는가"에 대한 요구사항 정의서이지, 코드 스타일 강제 문서가 아니다.
4. `rooms`, `hotels`, `bookings` 등 이미 존재하는 Entity와 연관관계를 맺어야 하므로, 그 Entity들의 실제 필드명/타입을 먼저 확인한 뒤 아래 신규 Entity를 작성한다.
5. MyBatis 매퍼는 이미 만들어져 있으므로 절대 새로 만들지 않는다. 신규 기능은 JPA로 구현한다.

## 1. 기능 개요

호텔 하나당 다음 두 가지를 관리하는 CRUD 기능을 만든다.

**A. 시즌별 요금**: 객실 유형(스탠다드/디럭스/스위트/프리미엄) 각각에 대해, 시즌(성수기/준성수기/평시/비수기 등, 개수 제한 없음)을 여러 개 등록할 수 있고, 시즌마다 평일요금과 주말요금을 가진다.

**B. 요금 정책**: 호텔 전체에 적용되는 정책 4가지 — 최소 투숙 기간, 체크인/체크아웃 시간, 취소 수수료, 유아/아동 요금.

화면은 요약 정보 영역, 객실 유형별 탭, 시즌별 요금 목록, 요일별 할증 정보, 요금 정책 4가지로 구성된다. (아래에서 각각 텍스트로 설명한다.)

---

## 2. 화면 구성 상세 (텍스트 설명)

### 2-1. 요약 정보 3가지
- 평균 객실 단가: 활성화된(is_active=true) 전체 객실의 base_price 평균값. 전월 대비 증감률(%)도 함께 계산해서 보여준다.
- 현재 시즌 요금: 오늘 날짜 기준으로 진행 중인 시즌명과 그 시즌의 스탠다드룸 평일요금. 비수기 대비 증감률(%)도 함께 계산한다.
- 다음 시즌 예정: 오늘 이후 가장 가까운 시작일을 가진 시즌의 시작일, 그날까지 남은 일수(D-day), 그 시즌이 몇 개 객실유형에 적용되는지.
- 요금 경쟁력 지수, 요금 시뮬레이터는 만들지 않는다 (요청사항으로 제외 확정).

### 2-2. 객실 유형 탭
스탠다드 / 디럭스 / 스위트 / 프리미엄 4개 탭. 탭을 바꾸면 아래 시즌 목록과 요일별 할증 정보가 그 유형 기준으로 바뀐다.

### 2-3. 시즌별 요금 목록
선택된 객실 유형에 등록된 시즌들을 리스트로 보여준다. 각 항목은 시즌명, 시작일~종료일, 평일 기준요금, 주말요금, 상태(예정/진행중/종료)를 가지며, 수정/삭제가 가능하다. 신규 시즌 등록 버튼도 이 영역에 있다.

### 2-4. 시즌 신규 등록/수정 폼
아래 항목을 한 화면(또는 한 모달)에서 입력받아 저장한다.
- 객실 유형 선택 (스탠다드/디럭스/스위트/프리미엄 중 1개)
- 시즌명 (텍스트)
- 시작일, 종료일 (날짜)
  - 상태(예정/진행중/종료)는 사용자가 입력하지 않고, 오늘 날짜와 시작일·종료일을 비교해서 서버가 자동으로 계산한다.
  - 같은 호텔·같은 객실유형 안에서 기존 시즌 기간과 겹치면 저장을 막고 어느 시즌과 겹치는지 알려준다.
- 평일 기준요금 (숫자, 필수)
- 주말요금 (숫자, 선택)
  - 비워두면 평일요금의 1.2배(20% 할증)로 자동 계산한다.
  - 값을 입력하면 그 값을 그대로 사용한다.
- 요일별 요금 정책 적용 여부 (on/off)
  - on이면: 주말요금 직접입력 대신, "요일별 할증률"을 평일요금에 곱해서 주말 가격을 계산한다.
  - off면: 위에서 입력한 주말요금을 그대로 사용한다.
  - 두 방식은 동시에 쓰지 않는다 (on이면 주말요금 직접입력 필드는 비활성화).
- 최소 투숙 박수 (이 시즌에만 적용되는 최소 숙박일수)
- 저장 전에 최종 요일별 요금(평일가/주말가)이 어떻게 계산되는지 미리 계산해서 화면에 보여준다 (저장 API 호출 전 프론트에서 계산, 실제 저장은 별도 확정 버튼으로).

### 2-5. 요일별 할증 정보
평일/주말 2가지로만 구분한다. 주말(금/토/일)에는 동일한 할증률 하나가 적용되고, 평일(월~목)은 할증이 없다. (요일 하나하나 다른 비율을 주는 기능은 만들지 않는다.)

### 2-6. 요금 정책 4가지
호텔 전체 단위로 아래 4가지를 조회/수정할 수 있다.
1. 최소 투숙 기간: 시즌별로 다르게 설정 가능 (2-4에서 입력한 값을 시즌마다 따로 가짐).
2. 체크인/체크아웃 시간: 시:분 단위로 설정.
3. 취소 수수료 정책: 체크인 며칠 전까지는 수수료 없음(기본 3일), 그 기한이 지나면(당일 포함) 고정 수수료율 적용(기본 100%). 구간을 여러 단계로 나누는 기능은 만들지 않는다.
4. 유아/아동 요금: 무료 기준 나이와, 그 기준을 넘으면 무료로 할지 할인(%)으로 할지 선택. 할인일 때만 할인율을 입력받는다.

## 3. DB 스키마 — 최종 확정 (ERDCloud 반영 완료, 재논의 불필요)

### season_rates 테이블에 추가된 컬럼
- hotel_id : BIGINT, NOT NULL — hotels.hotel_id를 참조하는 FK
- min_stay_nights : INT, NOT NULL, 기본값 1 — 이 시즌의 최소 투숙 박수
- weekday_policy_enabled : BOOLEAN, NOT NULL, 기본값 TRUE — 요일별 요금 정책 on/off

### rate_policies 테이블 변경사항
- room_id 컬럼을 hotel_id로 변경함 (방 단위가 아니라 호텔 전체 단위 정책이기 때문)
- child_rate_type : ENUM('FREE','DISCOUNT'), NOT NULL, 기본값 'FREE'
- child_discount_rate : NUMERIC(5,2), NULL 허용 — child_rate_type이 DISCOUNT일 때만 사용
- 기존에 있던 취소 기한(일), 취소 수수료율 컬럼은 그대로 사용 (신규 테이블 만들지 않음)
- 얼리체크인 관련 컬럼은 만들지 않기로 확정함 (기능 자체 제외)

### weekday_rates 테이블
- 요일 컬럼은 ENUM('WEEKDAY','WEEKEND') 두 값만 가짐 (월~일 7개로 세분화하지 않음)

### 참고 (급하지 않은 사항)
- season_rates.hotel_id, rate_policies.hotel_id가 ERDCloud 상에는 BIGSERIAL로 되어있는데, 참조용 FK 컬럼이므로 BIGINT가 맞다. 우선순위 낮음, 여유 있을 때 수정.
- promotion.dis_value 타입이 STRING으로 되어있는데 MySQL 표준 타입이 아니므로 NUMERIC이나 INT로 바꿔야 함.
- users.membership 컬럼 정의 끝에 괄호가 하나 더 있어서 문법 오류임 (`))`).

## 4. 신규 Entity가 가져야 할 필드 (참고용 — 실제 클래스 작성은 0번 지침에 따라 기존 코드 스타일 확인 후 진행)

### 시즌 요금 정보 (season_rates 매핑)
필드: seasonRateId(PK), hotelId, roomType, seasonName, startDate, endDate, weekdayPrice, weekendPrice, status(UPCOMING/ONGOING/ENDED, 조회 시점마다 재계산), minStayNights, weekdayPolicyEnabled, policyId

### 요일별 할증 정보 (weekday_rates 매핑)
필드: weekdayRateId(PK), seasonRateId, dayType(WEEKDAY/WEEKEND), rateMultiplierPercent

### 요금 정책 (rate_policies 매핑)
필드: policyId(PK), hotelId, checkInTime, checkOutTime, cancelDeadlineDays, cancelFeeRate, freeChildAge, childRateType(FREE/DISCOUNT), childDiscountRate

각 필드에 대한 주석과 정확한 타입은 실제 Entity 파일 작성 시 위 3번 섹션 스키마를 그대로 따른다.

## 5. API 설계 (참고용 — 실제 URL 규칙은 기존 Controller 컨벤션 확인 후 맞출 것)

필요한 기능은 다음과 같다. 정확한 경로/메서드명은 기존 프로젝트의 API 네이밍 규칙에 맞춘다.

- 호텔의 요약 정보(평균단가, 현재시즌요금, 다음시즌예정) 조회

- 시즌 요금 수정
- 시즌 요금 삭제
- 시즌 요금 저장 전 요일별 최종가 미리보기 계산
- 호텔의 요금 정책 4가지 조회
- 최소 투숙 기간 수정
- 체크인/체크아웃 시간 수정
- 취소 수수료 정책 수정 (취소기한일수 + 수수료율 2개 값만)
- 유아/아동 요금 정책 수정

## 6. 핵심 계산 로직

1. 시즌 상태는 저장 시점이 아니라 조회 시점마다 오늘 날짜 기준으로 재계산한다. (오늘<시작일 → 예정, 시작일≤오늘≤종료일 → 진행중, 오늘>종료일 → 종료)
2. 시즌 겹침 검증: 같은 호텔·같은 객실유형 내에서 [시작일,종료일] 구간이 겹치면 저장을 막는다.
3. 주말요금 미입력 시: 주말요금 = 평일요금 × 1.2
4. 요일별 정책 on일 때 최종 요금: 평일(월~목)은 평일요금 그대로, 주말(금/토/일)은 평일요금 × (1 + 할증률/100). off일 때는 주말도 입력된 주말요금을 그대로 쓴다.
5. 평균 객실 단가: 활성 객실의 base_price 평균, 전월 대비 증감률 = (이번달 평균 - 저번달 평균) / 저번달 평균 × 100
6. 비수기 대비 증감률: (성수기 스탠다드 요금 - 비수기 스탠다드 요금) / 비수기 스탠다드 요금 × 100
7. 취소 수수료: (체크인일 - 취소요청일) ≥ 취소기한일수(기본 3일) → 수수료 0%. 그보다 짧으면(당일 포함) → 정해진 수수료율(기본 100%) 적용.

## 8. 작업 우선순위
1. 기존 코드 구조/컨벤션 파악 (0번 지침)
2. DB 스키마 반영 확인 (3번 섹션 기준으로 실제 DB에 적용됐는지 점검)
3. Entity/Repository 작성
4. 요금 정책 CRUD API
5. 시즌 요금 CRUD API (겹침검증, 자동계산 포함)
6. 요약 통계 API
7. 프론트 연결
