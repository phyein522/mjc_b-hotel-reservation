sales_analysis 기능 개발 중, 다른 팀원들의 실제 도메인 코드가 아직 없어서
임시로 Fake 접두사가 붙은 엔티티/레포지토리/서비스를 만들어 사용했다.
1차 머지가 완료되어 팀원들의 실제 코드 중 일부가 프로젝트에 들어왔다.
아래 기준에 따라, 실제 파일이 이미 존재하는 경우에만 대응하는 fake 파일을 삭제하고
import를 실제 경로로 교체한다. 실제 파일이 아직 없는 항목은 절대 건드리지 말고 그대로 둔다.


1단계 — 먼저 확인할 것 (판단 기준)

아래 표의 "실제 파일 예상 경로"에 실제로 파일이 존재하는지 프로젝트 전체를 검색해서 확인한다.


존재하면 → 2단계, 3단계 진행 (해당 fake 파일 삭제 + import 교체)
존재하지 않으면 → 아무것도 하지 말고 fake 파일을 그대로 유지한다.
이름이 100% 똑같지 않을 수 있으니(User, Member, UserEntity 등) 비슷한 이름의 실제 엔티티/레포지토리도 함께 찾아서 알려준다.


Fake 파일실제 파일이 있을 것으로 예상되는 위치com/mjc/hotel/user/entity/FakeUser.javacom/mjc/hotel/user/entity/*.java (예: User, Member 등)com/mjc/hotel/user/repository/FakeUserRepository.javacom/mjc/hotel/user/repository/*.javacom/mjc/hotel/sales_analysis/entity/FakeHotel.java호텔 도메인 담당자 패키지 내 Entitycom/mjc/hotel/sales_analysis/entity/FakeRoom.java객실 도메인 담당자 패키지 내 Entitycom/mjc/hotel/sales_analysis/entity/FakeBooking.java예약 도메인 담당자 패키지 내 Entitycom/mjc/hotel/sales_analysis/entity/FakePayment.java결제 도메인 담당자 패키지 내 Entitycom/mjc/hotel/sales_analysis/entity/FakePaymentStatus.java결제 상태 Enumcom/mjc/hotel/sales_analysis/entity/FakeChannel.java채널 테이블 Entitycom/mjc/hotel/sales_analysis/entity/FakeRoomType.java객실 타입 테이블 Entitycom/mjc/hotel/sales_analysis/repository/FakeHotelRepository.java호텔 Repositorycom/mjc/hotel/sales_analysis/repository/FakeRoomRepository.java객실 Repositorycom/mjc/hotel/sales_analysis/repository/FakeBookingRepository.java예약 Repositorycom/mjc/hotel/sales_analysis/repository/FakePaymentRepository.java결제 Repository


2단계 — 실제 파일이 확인된 fake 파일만 삭제

1단계에서 "실제 파일 존재 확인됨"으로 판정된 항목만 물리적으로 삭제한다.
삭제 전에 어떤 fake 파일을, 어떤 실제 파일로 교체하는지 목록으로 요약해서 먼저 보여준다.
(한 번에 다 지우지 말고, 확인 목록 → 승인 → 삭제 순서로 진행)


3단계 — import 및 참조 코드 교체

아래 두 파일이 핵심 교체 대상이다. fake import를 실제 경로로 바꾸고,
실제 엔티티의 필드명이 다르면(예: roomId vs id, number vs roomNumber) 그에 맞게 코드도 함께 수정한다.


com/mjc/hotel/sales_analysis/service/SalesAnalysisServiceImpl.java

Fake 엔티티/레포지토리 import → 실제 엔티티/레포지토리 import로 교체



com/mjc/hotel/sales_analysis/SalesAnalysisServiceTest.java

테스트 코드 내 Fake 관련 import 및 빌더(builder) 호출부 전부 실제 필드명 기준으로 교체






4단계 — 지켜야 할 협업 규칙 (수정 금지 원칙)


sales_analysis 패키지 외부(다른 팀원 도메인) 파일은 직접 수정하지 않는다. 만약 꼭 고쳐야 한다면 무엇을, 왜 고쳐야 하는지 먼저 설명하고 승인을 받는다.
다른 팀원의 Entity는 조회(read)만 하고 절대 변경하지 않는다.
SalesAnalysisServiceStub.java (Stub 모드용, spring.profiles.active=stub)는 삭제하지 않는다. 이건 실제 DB 없이 테스트할 때 계속 필요하다.
hotel/build.gradle, SecurityConfig.java 등 공용 설정 파일은 이번 작업 범위가 아니므로 건드리지 않는다.
