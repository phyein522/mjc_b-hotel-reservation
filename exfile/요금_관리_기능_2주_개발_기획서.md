# [기획서] 호텔 요금 및 요금 정책 관리 기능 개발 기획서 (2주 개발 로드맵)

본 기획서는 **호텔 요금(Rates) 관리 및 요금 정책(Policies) 기능**에 대한 요구사항 분석, 기존 코드 스타일 분석, 데이터베이스 물리 스키마 매핑 설계, API 상세 스펙 및 2주일(14일) 동안의 일자별 개발 일정을 다룹니다.

---

## 1. 기획 개요 및 개발 제약 조건

### 1.1 개발 목적
- 호텔별 시즌 요금(성수기/준성수기/평시/비수기 등) 관리 기능(CRUD) 구축
- 호텔 전체에 적용되는 요금 정책(최소 투숙 박수, 체크인/체크아웃, 취소 수수료, 유아/아동 요금) 설정 기능 구축
- 호텔 지표 대시보드를 위한 요금 요약 정보(평균 단가, 현재 시즌 정보, 다음 시즌 D-day) 기능 제공

### 1.2 개발 제약 조건 (⚠️ 절대 규칙)
1. **작업 범위의 엄격한 격리**: 
   - 새로 추가되는 모든 소스 코드(Entity, DTO, Controller, Service, Repository)는 반드시 **`com.mjc.hotel.rates` 패키지 하위**에만 위치시킨다.
   - `rates` 패키지 외부의 기존 파일(예: `rooms`, `hotels`, `bookings` 패키지 등)은 절대 수정, 삭제, 이동하지 않는다.
2. **연관 관계 처리**:
   - 기존 Entity(`HotelEntity`, `RoomEntity`)의 참조가 필요하더라도 해당 클래스 코드를 수정할 수 없다. `rates` 패키지 내 Entity에서 `@ManyToOne` 등으로 단방향 참조하거나, FK 값(`hotelId`, `roomId`)만 필드로 유지하여 간접 참조한다.
3. **기술 스택 일치**:
   - **Java 21**, **Spring Boot 4.1.0**, **Spring Data JPA**, **MySQL**, **Lombok** 적용.
   - 기존에 작성된 MyBatis 매퍼는 수정하거나 새로 작성하지 않고, 신규 기능은 순수 **Spring Data JPA**로만 구현한다.
4. **물리 스키마 준수**:
   - `hotel_erd (2).sql`에 정의된 한글 컬럼명과 물리 컬럼 사양을 완벽히 매핑해야 하므로 JPA `@Column(name = "...")` 속성을 통해 DB 스키마 형식을 그대로 따른다.

## 2. 요구사항 정의 및 비즈니스 로직

### 2.1 요약 정보 (Dashboard API)
- **평균 객실 단가**: 활성화된(`is_active = true`) 전체 객실의 `base_price` 평균값을 구한다. 추가로 전월 대비 증감률(%)을 함께 반환한다.
  - *증감률 공식*: `((이번달 평균 - 저번달 평균) / 저번달 평균) * 100`
- **현재 시즌 요금**: 오늘 날짜 기준으로 진행 중인 시즌의 이름과 해당 시즌의 스탠다드룸 평일 요금을 반환한다. 비수기 기준요금 대비 증감률(%)도 계산하여 반환한다.
- **다음 시즌 예정**: 오늘 이후 가장 가까운 시작일을 가진 시즌의 시작일, D-day(남은 일수), 그리고 해당 시즌이 적용되는 객실 유형 개수를 반환한다.

### 2.2 객실 유형 탭
- 객실 유형은 기존 시스템과 동일하게 **Standard(스탠다드) / Suite(스위트) / Deluxe(디럭스)** 3가지로만 구분한다. (프리미엄은 제외)
- 화면 탭 전환 시 선택한 객실 유형에 맞춰 시즌 요금 목록 및 요일별 할증 정보가 전환되어 노출된다.

### 2.3 시즌별 요금 CRUD 및 정책
- **시즌별 요금**: 시즌명, 시작일~종료일, 평일 요금, 주말 요금을 포함한다.
- **시즌 상태 자동 계산**:
  - `오늘 < 시작일` : **UPCOMING (예정)**
  - `시작일 <= 오늘 <= 종료일` : **ONGOING (진행중)**
  - `오늘 > 종료일` : **ENDED (종료)**
- **시즌 기간 중복 검증**:
  - 동일 호텔 및 동일 객실 유형 내에서 신규/수정하려는 시즌의 `[시작일, 종료일]` 기간이 기존 시즌의 기간과 하루라도 겹칠 경우 예외를 발생시키고, 어떤 시즌과 겹치는지 예외 메시지로 명확히 리턴한다.
- **주말요금 자동 계산**:
  - 주말요금을 사용자가 기입하지 않고 빈 값으로 두면 **평일 요금의 1.2배(20% 할증)**로 자동 계산되어 적용된다.
- **요일별 할증 정책 (On/Off)**:
  - `weekday_policy_enabled` 필드가 `true(ON)`이면, 사용자가 입력한 주말요금을 무시하고 **요일별 할증률(weekday_rates의 변동률)**을 평일 요금에 곱해서 최종 주말 가격을 계산한다.
  - `false(OFF)`이면 입력된 주말요금을 그대로 사용한다. (두 방식은 배타적으로 작동하며, ON인 경우 화면에서 주말요금 입력 칸은 비활성화된다.)
  - 요일별 할증은 평일(월~목, 할증 없음)과 주말(금~일, 단일 할증률 적용) 2종으로 단순하게 관리한다.

### 2.4 요금 정책 4가지 (Hotel-wide Policy)
호텔 전체 단위 정책으로, `rate_policies` 테이블에서 조회 및 수정할 수 한다.
1. **최소 투숙 기간**: 기본 1박. 시즌별로 다른 최소 숙박 박수를 설정할 수 있도록 설계한다 (`season_rates.min_stay_nights` 연동).
2. **체크인/체크아웃 시간**: 시:분 단위 (`HH:mm:ss` 포맷)로 설정하여 조회 및 수정한다.
3. **취소 수수료 정책**: 
  - 체크인 3일 전까지는 수수료 0% (취소 가능 기한 기본 3일).
  - 3일 이내 혹은 당일 취소 시 고정 수수료율 적용 (기본 100%). 다단계 취소 정책은 제외한다.
4. **유아/아동 요금 정책**:
  - 무료 기준 나이 설정 (예: 12세 이하 무료).
  - 초과 시 요금 부과 방식 설정: 무료(`FREE`) 또는 할인율 적용(`DISCOUNT`). 할인율일 경우에만 할인율(`%`)을 설정할 수 있다.

## 5. API 설계 안 (REST API Spec)

### 5.1 요약 정보 API
- **GET** `/api/rates/hotels/{hotelId}/summary`
  - 응답 데이터:
    ```json
    {
      "averageBasePrice": 125000.0,
      "priceChangeRate": 5.4,
      "currentSeasonName": "성수기",
      "currentStandardWeekdayPrice": 150000.0,
      "increaseRateComparedToOffSeason": 25.0,
      "nextSeasonStartDate": "2026-07-25",
      "nextSeasonDDay": 16,
      "nextSeasonRoomTypeCount": 2
    }
    ```

### 5.2 시즌별 요금 API
- **GET** `/api/rates/hotels/{hotelId}/rooms/{roomType}`
  - 특정 호텔의 특정 객실유형(RoomType: 1=Standard, 2=Suite, 3=Deluxe)에 매핑된 시즌별 요금 목록 조회.
- **POST** `/api/rates`
  - 시즌 요금 신규 등록 (동일 객실유형 기간 중복 검증 필수).
- **PUT** `/api/rates/{seasonRateId}`
  - 시즌 요금 상세 정보 수정 (중복 검증 포함).
- **DELETE** `/api/rates/{seasonRateId}`
  - 시즌 요금 삭제.
- **POST** `/api/rates/preview`
  - 저장 전 평일/주말 요금 및 요일별 정책에 따른 최종 계산 금액 미리보기 계산기.
  - 요청 데이터: `{ "weekdayPrice": 100000, "weekendPrice": null, "weekdayPolicyEnabled": true, "multiplier": 20.0 }`
  - 응답 데이터: `{ "calculatedWeekdayPrice": 100000.0, "calculatedWeekendPrice": 120000.0, "increaseRate": 20.0 }`

### 5.3 요금 정책 API
- **GET** `/api/rates/policies/hotels/{hotelId}`
  - 특정 호텔의 요금 정책 4가지 정보 조회.
- **PUT** `/api/rates/policies/hotels/{hotelId}/min-stay`
  - 최소 투숙 기간 수정.
- **PUT** `/api/rates/policies/hotels/{hotelId}/times`
  - 체크인/체크아웃 시간 수정.
- **PUT** `/api/rates/policies/hotels/{hotelId}/cancellation`
  - 취소 수수료 정책 수정.
- **PUT** `/api/rates/policies/hotels/{hotelId}/child-rates`
  - 유아/아동 요금 정책 수정.

## 7. 주요 테스트 및 예외 검증 시나리오

1. **시즌 중복 검증**:
   - 호텔 A의 디럭스 룸에 `2026-07-01 ~ 2026-07-15` 성수기 시즌이 이미 있을 때, 동일 룸 유형에 `2026-07-10 ~ 2026-07-20` 기간의 준성수기 시즌을 추가하려는 경우 -> **HTTP 400 Bad Request (시즌 중복 에러)** 반환 검증.
2. **시즌 자동 판단**:
   - `startDate`가 오늘 이후이면 조회 시 `status` 필드가 `UPCOMING`으로 올바르게 자동 연산되는가 검증.
3. **요금 자동 연산**:
   - 주말 요금 필드가 Null인 상태로 저장되었을 때 평일 요금 대비 1.2배로 설정되는지 검증.
   - `weekday_policy_enabled`가 true일 때 `weekday_rates` 테이블의 요금변동률이 최종 주말 요금에 정확히 적용되는지 검증.
4. **아동 요금 유효성**:
   - `childRateType`이 `DISCOUNT`임에도 `childDiscountRate`이 누락되거나 음수인 경우에 대한 Validation 예외 검증.
