# 호실 추가(Room CRUD) 기능 기획서

> ⚠️ **최우선 원칙**: `com.mjc.hotel.rates` 패키지 이외의 모든 폴더/파일은 **읽기(참조) 전용**입니다. 절대 수정·생성·삭제하지 않습니다. `rates` 패키지 내부에서만 코드를 작성/수정합니다.
>
> ⚠️ **패키지 관계 확정**: `com.mjc.hotel.rates`(새로 코드를 작성하는 곳)와 `com.mjc.hotel.rooms`(기존 `RoomEntity`, enum들이 있는 곳)는 **완전히 별개의 패키지**입니다. `rooms`는 참고/참조 전용이며 수정하지 않습니다.

> **한줄 요약**: 호텔 하나를 고르면 그 호텔에 딸린 "호실(객실)"을 관리자가 등록/수정/삭제할 수 있는 CRUD 화면과 API를, `rates` 패키지 안에서만 2주 안에 만드는 계획서입니다.
>
> **쉽게 말하면**: 호텔 사장님이 "우리 호텔에 이런 방이 있어요"라고 입력하는 화면 하나를 만드는 겁니다. 방 번호, 층수, 가격, 최대 인원, 편의시설(와이파이/티비 등), 사진, 설명을 입력하고 저장하면 손님들이 나중에 이 방을 볼 수 있게 되는 거예요. 단, 이 작업은 내 담당 폴더(`rates`) 안에서만 하고, 다른 팀원 폴더는 절대 손대지 않습니다.

## 0. 작업 범위 제약사항 (필수 준수 — 절대 원칙)

- **`com.mjc.hotel.rates` 이외의 폴더/파일은 일체 건드리지 않는다.** 조회·읽기(참조) 목적으로만 열어보고, 절대 수정·생성·삭제하지 않는다.
  - 여기에는 다른 팀원이 작업 중인 `hotels`, `bookings`, `payment`, `sales_analysis` 패키지는 물론, `com.mjc.hotel.rooms`(기존 RoomEntity가 있는 패키지)도 포함됨 — **참조만 가능, 수정 불가**.
  - 공용 엔티티(`HotelEntity`, `BaseEntity` 등)와 공용 테이블(`users`, `hotels`, `bookings` 등)도 마찬가지로 **참조만 가능, 코드/스키마 수정 절대 불가**.
- `com.mjc.hotel.rates` **패키지 내부는** 자유롭게 구조를 짜도 됨 — 다만 `rooms` 패키지의 기존 `RoomEntity`/enum 코드를 그대로 가져올지, 참고만 해서 새로 짤지는 위의 "확인 필요한 불일치" 항목이 정리된 후 결정.
- AGY(바이브 코딩 CLI)에게 작업을 넘길 때 프롬프트 맨 앞에 반드시 다음을 명시: **"`com.mjc.hotel.rates` 패키지 이외의 파일은 절대 수정하지 말 것 (`com.mjc.hotel.rooms` 포함). 다른 패키지는 참조만 허용."**

## 1. 기능 개요

- **대상 화면**: 호텔 관리자 대시보드 > 예약 관리 > **호실 추가** (첨부 와이어프레임 기준)
- **핵심 목적**: 특정 호텔에 속한 객실(호실) 매물을 등록/조회/수정/삭제(CRUD)하여, 이후 방문객이 예약할 수 있는 상품 단위를 만든다.
- **전제**: 호텔 선택 → 해당 호텔 컨텍스트 안에서 호실 CRUD 진행 (호실은 반드시 하나의 호텔에 종속)

## 2. 화면 구성 (와이어프레임 매핑)

| 영역 | 필드 | 비고 |
|---|---|---|
| 기본 정보 | 호실 번호(`number`), 층수(`floor`), 객실 타입(`roomType`: enum) | 객실 타입은 드롭다운(Standard/Suite/Deluxe/Premium) |
| 요금 및 수용 인원 | 기본 요금(`basePrice`), 최대 수용 성인(`maxAdult`), 최대 수용 어린이(`maxChild`) | 기본 요금은 1박 기준 |
| 편의 시설 및 특징 | Wi-Fi, TV, 욕조, 시티뷰, 오션뷰, 조식포함, 금연 | 토글 버튼(다중 선택) |
| 객실 사진 | 이미지 업로드(드래그 앤 드롭), 썸네일 리스트 | 다중 업로드 지원 |
| 상세 설명 | `description` (텍스트) | 자유 서술형 |
| 액션 | 취소 / 호실 저장 | 저장 시 유효성 검사 후 등록 |

## 3. 데이터베이스 설계 (실제 `RoomEntity` 코드 기준으로 갱신)

> 팀원이 이미 구현해 둔 실제 엔티티 코드(`com.mjc.hotel.rooms` 패키지, 참고용/참조전용)를 확인함. 기존 SQL ERD와 필드명/타입이 다르므로 아래는 **실제 코드 기준**으로 정리한 최신 버전. 새로 작성하는 코드는 `com.mjc.hotel.rates`에 넣되, 아래 구조/필드는 참고만 함.

### `rooms` (`RoomEntity`, 참고 위치: `com.mjc.hotel.rooms.dto` — 참조만, 수정 금지)
```
room_id(PK), name, number, floor, size,
base_price, max_adult, max_child, is_active,
room_type(enum RoomType), room_status(enum RoomStatus),
room_view_option(enum RoomViewOption), room_bed_option(enum RoomBedOption),
block_start_date, block_end_date,
hotel_id(FK → HotelEntity, ManyToOne)
+ BaseEntity 상속(created_at/updated_at 등 공통 필드 포함)
```
- 타입/전망/침대/상태 4개 필드가 전부 **자체 enum**으로 구현되어 있음 (숫자 코드가 아니라 자바 enum 타입) → 프론트 드롭다운 값도 이 enum 이름 기준으로 맞춰야 함.
  - `RoomType`: Standard(1) / Suite(2) / Deluxe(3) / **Premium(4)** — SQL ERD에는 없던 Premium이 추가되어 있음, 와이어프레임 드롭다운에도 반영 필요
  - `RoomStatus`: EnableReservation / DisableReservation (추정)
  - `RoomViewOption`: CityView / RiverView / MountainView / OceanView (추정)
  - `RoomBedOption`: Floor / DoubleBed / QueenBed (추정)
- `hotel_id`는 `HotelEntity`와 `@ManyToOne`으로 직접 연결되어 있고, `IRoom`/`IHotel` 인터페이스를 통해 `hotelId`(Long)와 `hotel`(객체)이 항상 같은 값을 갖도록 동기화하는 패턴을 씀 → 이 패턴을 그대로 따라야 다른 팀 코드와 어긋나지 않음.
- ⚠️ enum ↔ DB 정수값 매핑을 어떻게 처리하는지(`AttributeConverter` 존재 여부) AGY가 코드 짤 때 확인 필요.

### `room_amenities` (편의 시설, 1:1 매핑) — *실제 엔티티 미확인, ERD 기준 유지*
```
room_amenity_id, wifi, tv, bathtub, city_view, ocean_view,
breakfast_included, non_smoking, room_id
```
- 와이어프레임의 "Wi-Fi/TV/욕조/시티뷰/오션뷰/조식포함/금연" 7개 토글과 정확히 1:1 매칭됨.
- ⚠️ `RoomEntity`에 이미 `room_view_option`(오션뷰/시티뷰 등)이 존재하므로, 편의시설 쪽 `city_view`/`ocean_view`와 **중복/역할 정리** 필요 (뷰는 rooms 쪽 enum으로 통일하고 amenities에서는 제거하는 방향을 AGY와 상의 권장).

### `room_images` (객실 사진, 1:N 매핑) — *실제 엔티티 미확인, ERD 기준 유지*
```
room_image_id, file_name, path, size, store_name, ext, room_id
```
- 드래그 앤 드롭 업로드 시 여러 장 저장 가능하도록 room_id 기준 1:N 구조.

## 4. API 명세 (초안)

| Method | Endpoint | 설명 |
|---|---|---|
| POST | `/api/hotels/{hotelId}/rooms` | 호실 등록 (기본정보+요금+수용인원) |
| GET | `/api/hotels/{hotelId}/rooms` | 호텔별 호실 목록 조회 |
| GET | `/api/rooms/{roomId}` | 호실 상세 조회 (편의시설+이미지+설명 포함) |
| PUT | `/api/rooms/{roomId}` | 호실 정보 수정 |
| DELETE | `/api/rooms/{roomId}` | 호실 삭제(soft delete 권장: `is_active=false`) |
| POST | `/api/rooms/{roomId}/amenities` | 편의시설 등록/수정 (upsert) |
| POST | `/api/rooms/{roomId}/images` | 이미지 업로드 (multipart, 다중 파일) |
| DELETE | `/api/rooms/{roomId}/images/{imageId}` | 이미지 개별 삭제 |

**요청 예시 (호실 등록, 실제 엔티티 필드명 기준)**
```json
{
  "number": "101",
  "floor": 1,
  "roomType": "STANDARD",
  "roomViewOption": "OCEAN_VIEW",
  "roomBedOption": "QUEEN_BED",
  "basePrice": 150000,
  "maxAdult": 2,
  "maxChild": 1,
  "description": "한강뷰가 보이는 스탠다드 룸입니다.",
  "amenities": {
    "wifi": true, "tv": true, "bathtub": false,
    "breakfastIncluded": true, "nonSmoking": true
  }
}
```
- 참고: 설명 필드(`description`)는 실제 코드에서 아직 확인되지 않아, AGY 구현 시 `RoomEntity`에 실제로 있는지 재확인 필요.

## 5. 클래스/패키지 구조

이미 존재하는 것 (`com.mjc.hotel.rooms` — **참조만, 여기에 코드 작성/수정 금지**):
```
com.mjc.hotel.rooms/
 ├─ enums/
 │   └─ RoomType.java, RoomStatus.java, RoomViewOption.java, RoomBedOption.java
 └─ dto/
     └─ RoomEntity.java   // BaseEntity 상속 + IRoom 인터페이스 구현, HotelEntity와 ManyToOne
```

새로 채워야 할 것 (`com.mjc.hotel.rates` — **실제 작업 위치**, 직관적 네이밍 원칙 적용):
```
com.mjc.hotel.rates/
 ├─ controller/
 │   ├─ RoomController.java          // 호실 CRUD API 진입점
 │   └─ RoomImageController.java     // 이미지 업로드 전담
 ├─ service/
 │   ├─ RoomService.java             // 호실 등록/수정/삭제 비즈니스 로직
 │   ├─ RoomAmenityService.java      // 편의시설 upsert 로직
 │   └─ RoomImageService.java        // 이미지 저장/삭제 로직
 ├─ repository/
 │   ├─ RoomRepository.java
 │   ├─ RoomAmenityRepository.java
 │   └─ RoomImageRepository.java
 ├─ entity/
 │   ├─ RoomAmenityEntity.java       // 기존 RoomEntity 네이밍 컨벤션(XxxEntity) 그대로 따름
 │   └─ RoomImageEntity.java
 └─ dto/
     ├─ RoomCreateRequest.java       // 등록 요청 DTO
     ├─ RoomUpdateRequest.java       // 수정 요청 DTO
     └─ RoomDetailResponse.java      // 상세 조회 응답 DTO (편의시설+이미지 포함)
```
- 클래스명은 "역할이 이름만 봐도 보이게" 원칙으로: `RoomService`(O) / `RService`(X), `RoomImageController`(O) / `ImgCtrl`(X)
- 기존 `RoomEntity`가 `XxxEntity` 네이밍과 `IRoom` 인터페이스 패턴을 쓰고 있으므로, `rates` 패키지에 새로 만드는 `RoomAmenityEntity`/`RoomImageEntity`도 같은 패턴(인터페이스 + Entity 접미사)을 따르는 걸 권장.
- `rates`와 `rooms`는 별개 패키지로 확정됨 — `rates`에서는 `rooms.RoomEntity`를 참고해서 동일한 필드/enum 구조로 새로 작성(또는 필요시 import해서 참조)하되, `rooms` 패키지 자체는 수정하지 않음.

## 7. 팀 통합(머지) 시 유의사항

- `hotel_id` FK는 이미 `RoomEntity`에 `@ManyToOne(HotelEntity)`로 구현되어 있고, `IRoom`/`IHotel` 인터페이스로 `hotelId`↔`hotel` 동기화 처리가 되어 있음 → 새로 만드는 코드도 이 동기화 패턴을 반드시 재사용 (직접 필드만 세팅하면 동기화 깨질 수 있음).
- 이미지 업로드 로직은 `hotel_images`에 이미 유사 구조(`filename, size, ext, storeName, path`)가 있으므로, 해당 담당 팀원의 업로드 구현체(스토리지 연동 방식: 로컬/S3 등)를 그대로 재사용하는 것이 중복 개발을 막는 길입니다.
- `roomType`/`roomStatus`/`roomViewOption`/`roomBedOption`은 이미 자체 enum으로 구현되어 있으므로, 편의시설(`room_amenities`)의 `city_view`/`ocean_view`와 `roomViewOption`의 역할이 겹치지 않도록 정리 필요 (겹치는 부분은 병합 전 팀 논의 대상).
- enum ↔ DB 정수값 변환(`AttributeConverter` 등) 방식이 팀 컨벤션으로 이미 있는지 확인 후, `RoomAmenity`/`RoomImage`에도 동일 방식 적용.

```
- [최우선] com.mjc.hotel.rates 패키지 이외의 모든 폴더/파일은 절대 수정하지 말 것
  (com.mjc.hotel.rooms, hotels, bookings, payment, sales_analysis 등 전부 포함).
  다른 패키지와 공용 엔티티는 참조(읽기)만 허용.
- 작업 범위: com.mjc.hotel.rates 패키지 내부에서만 코드 작성/수정
- 참고 자료: com.mjc.hotel.rooms의 RoomEntity, RoomType/RoomStatus/RoomViewOption/RoomBedOption enum
  (구조 참고용, 이 패키지 자체는 수정 금지)
- 대상 테이블: rooms(RoomEntity 기준), room_amenities, room_images
- 구현 대상: 호실 CRUD API + 관리자용 "호실 추가" 화면(첨부 와이어프레임 기준)

- 이미지 업로드는 드래그 앤 드롭 지원, 다중 파일
- 코드에는 핵심 로직마다 짧은 주석 포함
```

## 9. 추가로 저에게 넘겨주시면 좋은 파일 (필수는 아니지만 정확도를 높여줌)

1. ~~`room_type`/`bed_option`/`view_option` 코드값~~ → **완료** (RoomEntity, RoomType enum 코드로 확인함)
2. **`RoomAmenity`, `RoomImage` 엔티티 코드** (팀원이 이미 구현했다면) — 없다면 AGY가 `RoomEntity` 스타일 그대로 새로 만들면 됨
3. **기존에 구현된 `hotel_images` 업로드 로직 코드** (있다면) — 이미지 업로드 재사용 여부 판단용
4. **`com.mjc.hotel.rooms` 패키지의 실제 디렉토리 구조** (또는 `tree` 출력) — 파일 배치 확인용
