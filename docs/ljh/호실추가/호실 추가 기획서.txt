# 호실 추가(Room CRUD) 기능 기획서

> **한줄 요약**: 호텔 하나를 고르면 그 호텔에 딸린 "호실(객실)"을 관리자가 등록/수정/삭제할 수 있는 CRUD 화면과 API를 2주 안에 만드는 계획서입니다.
>
> **쉽게 말하면**: 호텔 사장님이 "우리 호텔에 이런 방이 있어요"라고 입력하는 화면 하나를 만드는 겁니다. 방 번호, 층수, 가격, 최대 인원, 편의시설(와이파이/티비 등), 사진, 설명을 입력하고 저장하면 손님들이 나중에 이 방을 볼 수 있게 되는 거예요.

---

## 0. 작업 범위 제약사항 (필수 준수)

- **`rates` 폴더 이외의 파일은 절대 수정 금지.** 팀원들이 각자 `hotels`, `bookings`, `payment`, `sales_analysis` 등 다른 패키지를 작업 중이므로, 이번 작업은 `rates`(호실/객실 관련) 패키지 내부에서만 이뤄져야 함.
- 공용 테이블(`users`, `hotels`, `bookings` 등)은 **읽기(참조)만** 하고, 스키마 변경이나 코드 수정은 하지 않음.
- AGY(바이브 코딩 CLI)에게 작업을 넘길 때도 "이 폴더 밖은 건드리지 마라"는 지시를 프롬프트에 명시해야 함.

---

## 1. 기능 개요

- **대상 화면**: 호텔 관리자 대시보드 > 예약 관리 > **호실 추가** (첨부 와이어프레임 기준)
- **핵심 목적**: 특정 호텔에 속한 객실(호실) 매물을 등록/조회/수정/삭제(CRUD)하여, 이후 방문객이 예약할 수 있는 상품 단위를 만든다.
- **전제**: 호텔 선택 → 해당 호텔 컨텍스트 안에서 호실 CRUD 진행 (호실은 반드시 하나의 호텔에 종속)

---

## 2. 화면 구성 (와이어프레임 매핑)

| 영역 | 필드 | 비고 |
|---|---|---|
| 기본 정보 | 호실 번호(`room_number`), 층수(`floor`), 객실 타입(`room_type`) | 객실 타입은 드롭다운(스위트/디럭스/스탠다드 등) |
| 요금 및 수용 인원 | 기본 요금(`base_price`), 최대 수용 성인(`max_adult`), 최대 수용 어린이(`max_children`) | 기본 요금은 1박 기준 |
| 편의 시설 및 특징 | Wi-Fi, TV, 욕조, 시티뷰, 오션뷰, 조식포함, 금연 | 토글 버튼(다중 선택) |
| 객실 사진 | 이미지 업로드(드래그 앤 드롭), 썸네일 리스트 | 다중 업로드 지원 |
| 상세 설명 | `description` (텍스트) | 자유 서술형 |
| 액션 | 취소 / 호실 저장 | 저장 시 유효성 검사 후 등록 |

---

## 3. 데이터베이스 설계 (기존 ERD 기준, 신규 컬럼 없이 재사용)

이미 ERD에 존재하는 3개 테이블을 그대로 활용합니다 (스키마 변경 불필요, 확인만 필요):

### `rooms` (호실 기본 정보 + 요금 + 수용인원 + 설명)
```
room_id, room_number, floor, status, name, size,
base_price, max_adult, max_children, is_active,
호텔아이디(hotel_id), bed_option, view_option, room_type,
description, block_start_date, bloack_end_date,
created_at, updated_at
```
- ⚠️ 팀 확인 필요: `호텔아이디` 컬럼명이 한글로 되어있음 → 통일성 위해 `hotel_id`로 리네이밍 협의 권장 (단, `rates` 폴더 밖 스키마이므로 임의 변경 금지, 팀 회의 안건으로 상정)
- ⚠️ `bloack_end_date` 오타로 보임 → 팀 컨벤션 확인 후 `block_end_date`로 수정 제안

### `room_amenities` (편의 시설, 1:1 매핑)
```
room_amenity_id, wifi, tv, bathtub, city_view, ocean_view,
breakfast_included, non_smoking, room_id
```
- 와이어프레임의 "Wi-Fi/TV/욕조/시티뷰/오션뷰/조식포함/금연" 7개 토글과 정확히 1:1 매칭됨.

### `room_images` (객실 사진, 1:N 매핑)
```
room_image_id, file_name, path, size, store_name, ext, room_id
```
- 드래그 앤 드롭 업로드 시 여러 장 저장 가능하도록 room_id 기준 1:N 구조.

## 4. API 명세 (초안)

| Method | Endpoint | 설명 |
|---|---|---|
| POST | `/api/hotels/{hotelId}/rooms` | 호실 등록 (기본정보+요금+수용인원) |
| GET | `/api/hotels/{hotelId}/rooms` | 호텔별 호실 목록 조회 |
| GET | `/api/rooms/{roomId}` | 호실 상세 조회 (편의시설+이미지+설명 포함) |
| PUT | `/api/rooms/{roomId}` | 호실 정보 수정 |
| DELETE | `/api/rooms/{roomId}` | 호실 삭제(soft delete 권장: `is_active=false`) |
| POST | `/api/rooms/{roomId}/amenities` | 편의시설 등록/수정 (upsert) |
| POST | `/api/rooms/{roomId}/images` | 이미지 업로드 (multipart, 다중 파일) |
| DELETE | `/api/rooms/{roomId}/images/{imageId}` | 이미지 개별 삭제 |

**요청 예시 (호실 등록)**
```json
{
  "roomNumber": "101",
  "floor": 1,
  "roomType": 2,
  "basePrice": 150000,
  "maxAdult": 2,
  "maxChildren": 1,
  "description": "한강뷰가 보이는 스탠다드 룸입니다.",
  "amenities": {
    "wifi": true, "tv": true, "bathtub": false,
    "cityView": true, "oceanView": false,
    "breakfastIncluded": true, "nonSmoking": true
  }
}
```

## 5. 클래스/패키지 구조 제안 (직관적 네이밍)

```
rates/
 ├─ controller/
 │   ├─ RoomController.java          // 호실 CRUD API 진입점
 │   └─ RoomImageController.java     // 이미지 업로드 전담
 ├─ service/
 │   ├─ RoomService.java             // 호실 등록/수정/삭제 비즈니스 로직
 │   ├─ RoomAmenityService.java      // 편의시설 upsert 로직
 │   └─ RoomImageService.java        // 이미지 저장/삭제 로직
 ├─ repository/
 │   ├─ RoomRepository.java
 │   ├─ RoomAmenityRepository.java
 │   └─ RoomImageRepository.java
 ├─ entity/
 │   ├─ Room.java
 │   ├─ RoomAmenity.java
 │   └─ RoomImage.java
 └─ dto/
     ├─ RoomCreateRequest.java       // 등록 요청 DTO
     ├─ RoomUpdateRequest.java       // 수정 요청 DTO
     └─ RoomDetailResponse.java      // 상세 조회 응답 DTO (편의시설+이미지 포함)
```
- 클래스명은 "역할이 이름만 봐도 보이게" 원칙으로: `RoomService`(O) / `RService`(X), `RoomImageController`(O) / `ImgCtrl`(X)

## 7. 팀 통합(머지) 시 유의사항

- `hotel_id`(또는 `호텔아이디`) 참조는 `hotels` 테이블에 FK로 걸려있어야 하나, 현재 ERD에는 `rooms`-`hotels` FK 제약이 명시되어 있지 않음 → 병합 전 팀에 FK 추가 여부 확인 필요.
- 이미지 업로드 로직은 `hotel_images`에 이미 유사 구조(`filename, size, ext, storeName, path`)가 있으므로, 해당 담당 팀원의 업로드 구현체(스토리지 연동 방식: 로컬/S3 등)를 그대로 재사용하는 것이 중복 개발을 막는 길입니다.
- `room_type`, `bed_option`, `view_option`은 ERD상 `INT` + 주석(스위트/디럭스 등)으로만 되어있어 실제 코드 매핑 테이블이 없음 → 프론트 드롭다운과 값이 어긋나지 않도록 Enum 또는 코드 테이블 확정 필요.

## 8. AGY(바이브 코딩)에게 넘길 프롬프트 요약

```
- 작업 범위: rates 폴더 내부만 수정. 다른 폴더는 절대 건드리지 말 것.
- 대상 테이블: rooms, room_amenities, room_images (스키마는 첨부 ERD 그대로 사용, 변경 금지)
- 구현 대상: 호실 CRUD API + 관리자용 "호실 추가" 화면(첨부 와이어프레임 기준)
- 클래스명은 역할이 드러나게 직관적으로 (RoomController, RoomService 등)
- 이미지 업로드는 드래그 앤 드롭 지원, 다중 파일
- 코드에는 핵심 로직마다 짧은 주석 포함
```

## 9. 추가로 저에게 넘겨주시면 좋은 파일 (필수는 아니지만 정확도를 높여줌)

1. **`rates` 폴더의 실제 디렉토리 구조** (또는 `tree` 출력) — 기존 컨벤션에 맞춰 파일을 배치하기 위함
2. **기존에 구현된 `hotel_images` 업로드 로직 코드** (있다면) — 이미지 업로드 재사용 여부 판단용
3. **`room_type` / `bed_option` / `view_option`의 실제 코드값 정의** (Enum이나 코드 테이블이 이미 있다면)
4. **프론트엔드 기존 코드/컴포넌트 구조** (React/Vue 등, 라우팅 방식 포함) — 화면을 어디에 어떻게 붙일지 결정하기 위함
5. **호텔 조회/선택 관련 기존 API 명세** (호실 등록 전 호텔 컨텍스트를 어떻게 받아오는지)
6. 팀에서 사용 중인 **코딩 컨벤션 문서**(있다면) — 네이밍/패키지 규칙 통일용

위 파일들이 없어도 기획은 진행 가능하지만, 있으면 AGY가 팀 기존 코드와 더 잘 맞물리게 작업할 수 있습니다.
