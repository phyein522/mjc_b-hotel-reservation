# [기획서] 호텔 호실 추가 및 관리 기능 개발 기획서 (2주 개발 로드맵)

본 기획서는 **호텔 호실(Room) 추가 및 관리 기능**에 대한 요구사항 분석, 와이어프레임 UI 구성 매핑, 데이터베이스 물리 스키마 연동 설계, REST API 스펙 및 2주일(10 영업일) 동안의 일자별 상세 개발 일정을 다룹니다.

---

## 1. 기획 개요 및 개발 제약 조건

### 1.1 개발 목적
- 특정 호텔 사장님(관리자)이 자사 호텔에 속한 호실(객실) 정보를 시스템에 등록, 조회, 수정, 삭제(CRUD)할 수 있는 관리자 화면과 API를 제공합니다.
- 호실 번호, 층수, 객실 타입, 1박 기본 요금, 최대 수용 인원(성인/어린이), 편의시설(와이파이, TV, 욕조, 뷰, 조식, 금연 등), 객실 사진(다중 업로드), 상세 설명을 체계적으로 저장하여 고객 예약 서비스의 상품 단위 데이터를 구축합니다.

### 1.2 개발 제약 조건 (⚠️ 팀 협업 필수 준수사항)
1. **작업 범위의 엄격한 격리**:
   - 수정 및 신규 개발은 **`com.mjc.hotel.rooms`** 및 **`com.mjc.hotel.room_images`** 패키지 내부로 한정합니다.
   - 타 패키지(`hotels`, `bookings`, `payment`, `sales_analysis` 등) 파일은 절대 직접 수정하거나 삭제하지 않습니다.
2. **공통 엔티티 및 인터페이스 패턴 준수**:
   - `HotelEntity` 등 타 패키지 엔티티는 `@ManyToOne` 단방향 또는 `hotelId`(FK) 참조로만 활용합니다.
   - 기존 팀 컨벤션인 `IRoom`, `IRoomImage` 인터페이스 및 `copyMembers()` 동기화 패턴을 그대로 재사용하여 데이터 일관성을 유지합니다.
3. **기술 스택**:
   - **Backend**: Java 21, Spring Boot 3.x / 4.x, Spring Data JPA, Lombok
   - **Frontend**: HTML5, Vanilla CSS, JavaScript (Fetch API / Drag & Drop File API)
   - **Database**: MySQL (`rooms`, `room_amenities`, `room_images`)

## 4. API 명세서

### 4.1 전체 API 목록 Summary

| Method | Endpoint | Description | Query/Body Param |
| :--- | :--- | :--- | :--- |
| **POST** | `/api/v1/room` | 호실 기본 정보 저장 | `RoomDto` JSON |
| **POST** | `/api/v1/room/image` | 호실 정보 + 이미지 파일 동시 업로드 저장 | Multipart (`requestDto` JSON + `files` List) |
| **POST** | `/api/v1/room/{roomId}/amenities` | 호실 편의시설 등록/수정 (Upsert) | `RoomAmenityDto` JSON |
| **GET** | `/api/v1/room/{roomId}` | 호실 단건 기본 정보 조회 | Path `roomId` |
| **GET** | `/api/v1/room/details/{roomId}` | 호실 상세 정보 조회 (편의시설 + 이미지 목록 포함) | Path `roomId` |
| **GET** | `/api/v1/room/hotel/{hotelId}` | 특정 호텔의 전체 호실 페이징 목록 조회 | Path `hotelId`, Pageable (`page`, `size`, `sort`) |
| **PATCH** | `/api/v1/room` | 호실 정보 부분 수정 | `RoomDto` JSON |
| **DELETE** | `/api/v1/room/{roomId}` | 호실 삭제 (Soft Delete / Hard Delete) | Path `roomId` |
| **DELETE** | `/api/v1/room-images/{roomImageId}` | 특정 객실 이미지 삭제 | Path `roomImageId` |

### 4.2 주요 API 상세 요청/응답 예시

#### [POST] `/api/v1/room/image` (호실 추가 저장 - 와이어프레임 대응 핵심 API)
- **Content-Type**: `multipart/form-data`
- **Request Part 1**: `requestDto` (application/json)
```json
{
  "hotelId": 1,
  "number": "101",
  "floor": 1,
  "roomType": "STANDARD",
  "roomStatus": "ENABLE_RESERVATION",
  "roomViewOption": "OCEAN_VIEW",
  "roomBedOption": "QUEEN_BED",
  "basePrice": 150000,
  "maxAdult": 2,
  "maxChild": 1,
  "size": 32,
  "description": "한강뷰와 바다 전경이 내려다보이는 쾌적한 스탠다드 객실입니다.",
  "amenity": {
    "wifi": true,
    "tv": true,
    "bathtub": false,
    "cityView": false,
    "oceanView": true,
    "breakfastIncluded": true,
    "nonSmoking": true
  }
}
```
- **Request Part 2**: `files` (MultipartFile List) - 이미지 파일 첨부

- **Response Body**:
```json
{
  "code": "INSERT_OK",
  "message": "ok",
  "data": {
    "roomDto": {
      "roomId": 10,
      "hotelId": 1,
      "number": "101",
      "floor": 1,
      "roomType": "STANDARD",
      "basePrice": 150000,
      "maxAdult": 2,
      "maxChild": 1,
      "description": "한강뷰와 바다 전경이 내려다보이는 쾌적한 스탠다드 객실입니다.",
      "isActive": true
    },
    "roomImages": {
      "content": [
        {
          "roomImageId": 1,
          "fileName": "room_view.jpg",
          "storeName": "uuid_room_view.jpg",
          "path": "/upload/rooms/10/",
          "size": 1048576,
          "ext": "jpg"
        }
      ]
    }
  }
}
```

## 7. 팀 통합(Merge) 및 엣지 케이스 처리 유의사항

1. **`hotelId` 동기화 패턴 유지**:
   - `RoomEntity`는 `hotelId`를 직접 갖고 있으며 타 패키지의 `HotelEntity`를 다룰 때 직접적인 변경을 가하지 않도록 주의합니다.
2. **이미지 파일 저장소 공유**:
   - 이미지 저장 경로는 `FileUtil`을 통해 일관되게 관리하며, 호실 삭제 시 DB 레코드뿐만 아니라 저장된 실물 이미지 파일도 `fileUtil.deleteFile()`로 함께 제거합니다.
3. **예외 처리 규칙**:
   - 존재하지 않는 `roomId` 또는 `hotelId` 조회 시 `ResponseCode` 기준 에러 응답 반환.
   - 필수 필드(`number`, `floor`, `roomType`, `basePrice`) 누락 시 400 Bad Request 에러 메시지 반환.
