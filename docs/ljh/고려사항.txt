[매출 분석 기능 설계 요청]

기술 스택

* Java 21
* Gradle
* Spring Boot 4.1.0 (Web, Security)
* MySQL
* Spring Data JPA + MyBatis
* Swagger UI
* Lombok

패키지 구조
com.mjc.hotel
├── advice
├── config
├── sales_analysis
│    ├── controller
│    ├── service
│    ├── repository
│    ├── entity
│    └── dto
└── student

현재 나는 sales_analysis(매출 분석) 기능만 담당하고 있다.

구현 목표

* 호텔 선택 시 해당 호텔의 "이번 달" 데이터를 기본으로 조회
* 이전 달부터 전년도 1월까지 월별 조회 가능
* 상단 월 버튼 클릭 시 해당 월의 모든 통계 조회

대시보드 항목

* 이번 달 총 매출
* 전월 대비 매출 증감률(%)
* 목표 매출
* 객실 점유율 및 전월 대비 증감률
* 오늘 객실 점유율
* 평균 객단가(ADR)
* 예약 건수 및 전월 대비 증감률
* 재방문율 및 전월 대비 증감률
* VIP 고객 재방문율

추가 기능

* 최근 7개월 객실 매출 막대 그래프
* 객실별 매출, 객실 수, 총 객실 매출
* 채널별 매출 비중(예약 앱별 %)
* 매출 TOP5 예약 내역

  * 고객명
  * 객실명
  * 결제 금액
  * 숙박 일수

중요 조건

* sales_analysis 폴더 내부에서만 작업
* 다른 패키지 수정 시 반드시 이유 설명
* 완성도보다 협업과 머지 충돌 방지를 우선 고려
* DTO 기반 설계
* 인터페이스 기반 설계
* Repository와 Service를 기능별로 분리
* 다른 팀원이 아직 구현하지 않은 기능은 Stub/TODO 처리
* 다른 테이블의 데이터를 읽어와 합산 후 HTML에 출력하는 구조로 설계

요청 사항

* 머지 충돌이 최소화되는 패키지 구조 설계
* Controller, Service, Repository 구조 설계
* API 설계
* DTO 설계
* 향후 팀원 기능과 쉽게 연동할 수 있는 구조 제안

중요
- 현재는 완성된 기능보다 협업과 머지 안정성을 우선한다.
- 다른 팀원의 기능이 추가되더라도 기존 sales_analysis 코드를 최대한 수정하지 않는 구조를 목표로 한다.
- 인터페이스와 DTO를 활용하여 향후 구현체 교체가 가능하도록 설계한다.

실제 데이터가 없어도 테스트 가능한 더미 데이터 구조를 제공할 것

데이터가 없는 경우
- 값은 null 대신 0 또는 "-" 로 반환
- API 에러가 아닌 정상 응답으로 처리

협업 조건
- 다른 팀원의 Entity를 직접 수정하지 말 것
- 필요한 데이터는 조회만 수행
- Entity 변경이 발생해도 DTO는 유지되도록 설계

프론트 처리
- 그래프는 프론트에서 그림
- 백엔드는 숫자 데이터만 반환

API 경로는 초기에 확정하고 이후 변경을 최소화할 것

예시
GET /api/sales/dashboard
GET /api/sales/monthly
GET /api/sales/channels
GET /api/sales/top-bookings
GET /api/sales/rooms
GET /api/sales/chart
